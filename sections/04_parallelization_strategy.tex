\chapter{Parallelization Methods}

\section{Task-Parallel YOLO Inference}

The program does not split the internal layers of the YOLO neural network. Instead, it parallelizes the repeated inference calls produced by video processing. Each MPI rank receives a subset of frame-tile tasks, runs YOLO locally, and returns person detections for global merging. Therefore, the level of parallelism is \textbf{task parallelism}.

The decomposition technique is a \textbf{hybrid temporal-spatial task decomposition}. The temporal dimension decomposes the video into frames, while the spatial dimension decomposes each frame into image tiles. This is not a recursive or speculative decomposition; each task is an independent YOLO inference call on one tile of one frame.

\subsection{Frame and Tile Tasks}

Let the input video contain $N$ frames. Each frame can be split into an $R \times C$ grid of tiles. The number of inference tasks is

\[
|\mathcal{T}| = N \times R \times C.
\]

Each task is one YOLO inference on one tile of one frame. A small grid such as $2 \times 2$ preserves more visual context and usually gives detector behavior closer to the original-frame baseline. A larger grid such as $5 \times 4$ creates more parallel tasks, but it also creates more tile boundaries, more duplicated detections, and more work for the final merge stage.

\subsection{Mapping Mode 1: Rank Mapping}

The first mapping mode is \texttt{--mapping rank}. All frame-tile tasks are flattened into a one-dimensional task list. For a frame id $f$ and tile id $t$, where each frame has $T = R \times C$ tiles, the task id is

\[
q = fT + t.
\]

With $P$ MPI ranks, tasks are assigned in a round-robin/block-cyclic style:

\[
owner(q) = q \bmod P.
\]

This mapping is simple and has low scheduling overhead. However, tiles from the same frame may be processed on different physical machines. As a result, many raw bounding boxes must be transferred to rank 0, and rank 0 must perform most of the final merging work.

\subsection{Mapping Mode 2: Weighted-Node Mapping}

The second mapping mode is \texttt{--mapping weighted-node}. This mode is used to better match the heterogeneous three-machine cluster. Instead of distributing every tile directly across all ranks, complete frames are first assigned to physical machines according to machine weights. In the final configuration, the selected weights are

\[
master : node1 : node2 = 4 : 6 : 2.
\]

This means node1 receives the largest share of frames, the master receives a medium share, and node2 receives the smallest share. For every group of 12 frames, frames 0--3 are assigned to the master, frames 4--9 to node1, and frames 10--11 to node2. The pattern then repeats.

After a frame is assigned to a physical machine, only the ranks on that machine divide the tiles of that frame among themselves. This mapping has two advantages. First, faster machines receive more work. Second, most duplicated boxes from tiles of the same frame can be merged locally before crossing the network.

\subsection{Communication Strategy and Topology}

The communication pattern depends on the mapping mode.

With \texttt{--mapping rank}, the topology is a \textbf{centralized master-worker gather}. Worker ranks run local YOLO inference and send serialized detections and timing metrics to rank 0. Rank 0 converts tile-local coordinates back to full-frame coordinates, merges detections from all tiles, removes duplicates, applies global non-maximum suppression, and computes the final people count.

With \texttt{--mapping weighted-node}, the topology is \textbf{hierarchical}. Ranks on the same physical machine first send their detections to a node leader. The node leader performs a local merge for frames owned by that machine. Then only the smaller merged result is sent to rank 0. Rank 0 still performs the final global merge, but it receives fewer raw detections.

The implementation supports both blocking and non-blocking result transfer. The stable benchmark numbers reported in this report use one consistent benchmark set. The non-blocking gather branch is treated as an implementation optimization: it changes how messages are transferred, but it does not change YOLO inference, coordinate remapping, duplicate removal, or final counting.

\subsection{Bounding Box Remapping and Global Merge}

Each YOLO task produces person bounding boxes in tile-local coordinates. If a tile starts at position $(x_s, y_s)$ in the original frame, a detected box coordinate is converted as

\[
x^{global} = x^{local} + x_s,
\qquad
y^{global} = y^{local} + y_s.
\]

After coordinate conversion, detections from all tiles belonging to the same frame are merged. A person near a tile boundary may appear in two neighboring tiles, so duplicate removal is required. The final detections for frame $F_i$ are computed as

\[
B_i^{final} = NMS(B_i^{raw}, \theta),
\qquad
count(F_i) = |B_i^{final}|.
\]

This global merge step is the reason the problem is not purely embarrassingly parallel. YOLO inference tasks are independent, but correct frame-level counting requires all detections belonging to the same frame.

\subsection{Parallel Algorithm}

\begin{algorithm}[H]
\caption{MPI YOLO People-Counting Pipeline}
\begin{algorithmic}[1]
\State Split the input video into frames.
\State Split each frame into an $R \times C$ grid of image tiles.
\State Build the list of frame-tile inference tasks.
\State Assign tasks using either rank mapping or weighted-node mapping.
\For{each MPI rank in parallel}
\State Select the tasks assigned to this rank.
\For{each assigned frame-tile task}
\State Crop the tile from the original frame.
\State Run YOLO inference on the tile.
\State Keep only person-class detections.
\State Store bounding boxes and timing metrics.
\EndFor
\State Send local detections for merging.
\EndFor
\If{rank is a node leader}
\State Merge detections inside the physical node.
\State Send compact node-level detections to rank 0.
\EndIf
\If{rank is 0}
\For{each frame}
\State Convert tile-local boxes to global frame coordinates.
\State Merge boxes from all tiles of the frame.
\State Remove duplicated boxes near tile boundaries.
\State Apply global non-maximum suppression.
\State Count the remaining boxes as the number of people.
\EndFor
\State Write per-frame counts and runtime metrics.
\EndIf
\end{algorithmic}
\end{algorithm}

\section{Metrics and Load Balance}

The report records wall-clock runtime, compute-oriented runtime, per-rank compute time, communication time, and idle time. Wall-clock runtime is the main runtime used for input-size selection and speedup.

Speedup is computed as

\[
S_p = \frac{T_1}{T_p},
\]

where $T_1$ is the one-process wall-clock runtime and $T_p$ is the wall-clock runtime using $p$ MPI processes. Parallel efficiency is

\[
E_p = \frac{S_p}{p}.
\]

Load balance is measured with the idle-gap indicator

\[
idle_gap = \frac{T_{\max} - T_{\min}}{T_{\max}},
\]

where $T_{\max}$ and $T_{\min}$ are the maximum and minimum total rank times. A smaller idle gap means the ranks finish more evenly. In this project, the weighted $4/6/2$ configuration is used because it reduces idle gap on the heterogeneous three-machine cluster.
